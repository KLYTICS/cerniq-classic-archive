# Unified Export Platform

Date: 2026-04-06

## Goal

Standardize CERNIQ's team-facing document downloads behind a backend-first export contract so ALM reports, sample documents, board packages, and preview documents follow one product standard for metadata, filenames, watermarks, and download handling.

## Covered Document Types

| Kind | Audience | Source | Delivery |
| --- | --- | --- | --- |
| `alm_report` | internal | institution-level ALM analysis | server-generated PDF |
| `sample_report` | sample | charter-number sample generation | server-generated PDF with watermark |
| `alco_pack` | internal | completed portal job | server-generated PDF |
| `preview_report` | sample | public preview slug | server-generated PDF with watermark |

## Backend Contract

### Manifest shape

Every team-facing document manifest returns:

- `id`
- `kind`
- `language`
- `audience`
- `filename`
- `mimeType`
- `status`
- `downloadUrl`
- `generatedAt`
- `expiresAt`
- `watermark`
- `sourceInstitutionId`
- `sourceJobId`

### Endpoints

Institution exports:

- `GET /api/alm/:institutionId/exports`
- `GET /api/alm/:institutionId/report?lang=en|es`

Sample exports:

- `GET /api/alm/sample-report/:charterNumber/exports`
- `GET /api/alm/sample-report/:charterNumber?lang=en|es`
- Existing compatibility endpoint remains: `POST /api/alm/sample-report`

Preview exports:

- `GET /api/alm/previews/:slug/exports`
- `GET /api/alm/previews/:slug/report?lang=en|es`

Portal job exports:

- `GET /api/portal/jobs/:jobId/exports`
- Existing ALCO binary endpoint remains: `POST /api/portal/jobs/:jobId/alco-pack?lang=en|es`

## Rendering Rules

- Enterprise/team documents are backend-generated by default.
- Client-side PDF capture is no longer the primary path for ALM, portal, demo/embed, or preview downloads.
- Sample and preview documents carry explicit review-purpose watermarks.
- Filenames are bilingual and standardized by document kind, institution/source label, language, and generation date.
- Binary responses include document kind/language/audience headers for consistent downstream handling.

## Frontend Standard

- Shared manifest fetch + download logic lives in `frontend/lib/document-exports.ts`.
- Shared stateful loading/error/download behavior lives in `frontend/hooks/useDocumentExports.ts`.
- Shared UI entrypoint lives in `frontend/components/exports/DocumentExportButtons.tsx`.
- Current migrated surfaces:
  - `frontend/app/alm/layout.tsx`
  - `frontend/app/alm/page.tsx`
  - `frontend/app/demo/embed/page.tsx`
  - `frontend/app/preview/[institution]/page.tsx`
  - `frontend/app/portal/page.tsx`
  - `frontend/app/portal/reports/[id]/page.tsx`
  - `frontend/components/portal/ReportProgressWS.tsx`

## Known Gaps

- The portal WebSocket completion payload still carries report URLs directly; the UI now prefers the manifest endpoint after completion, but the socket payload itself is not yet normalized to the manifest shape.
- CSV/Excel/JSON analytics exports are intentionally out of the unified V1 document scope.
- Prospect dossier sample downloads under the admin path remain server-generated but are not yet migrated onto the shared manifest contract.
