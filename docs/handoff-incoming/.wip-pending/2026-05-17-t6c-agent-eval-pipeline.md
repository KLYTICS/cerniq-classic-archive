- 2026-05-17 — **T6.c closure: end-to-end agent-eval pipeline — scorecard CLI, manual-bless workflow, regression gate, and locked baselines for all 4 agent types.** Closes the deferred regression-detection capability that T6.b enabled but didn't wire. Before this commit, `test/agent-evals/runner/regression-gate.ts` existed but had never been invoked because (a) there was no scorecard CLI to produce a results.json, (b) the alm_decision baseline was a placeholder `{meanScore: 0, perCase: {}}`, (c) no test exercised the gate's regression semantics so a refactor could silently break it, and (d) no npm script wired any of it together. T6.c lands all four pieces with locked-in scores and a 6-test subprocess integration spec. **What landed:** (1) `backend-node/test/agent-evals/script-registry.ts` — explicit static enumeration of all 14 LLM scripts (10 alm-decision + 1 cfo-copilot + 1 committee-report + 2 risk-monitor) paired with their golden case files. Two helpers: `scriptsForAgent(agentId)` filters by agent, `knownAgentIds()` returns the registry-order list. Explicit enumeration instead of glob discovery so adding a script is a single-file diff and reviewers can see the full surface in one place. (2) `backend-node/scripts/agent-eval.ts` (~130 LOC) — scorecard CLI: loads each registered entry, runs `scoreOffline` (which now routes through the T6.b per-agent adapter + weight profile), groups results by agentId, writes `test/agent-evals/results/<agent-id-lowercased>.json`, prints a banner-style scorecard, exits 0 only if every script scored ≥ 0.8. Flags: `--agent <ID>` to filter, `--json` for machine-readable stdout, `--quiet` for exit-code-only CI use, `--help`. (3) `backend-node/scripts/agent-eval-bless.ts` (~90 LOC) — manual bless tool: reads `results/<agent>.json`, writes `baselines/<agent>.json` with `{agentId, meanScore, perCase: {caseId: score}, updatedAt, note}` matching the existing `EvalBaseline` shape. Requires `BLESS_REASON` env var so every baseline change carries a rationale (Vol3 §Layer 2 Consistency gate). Per-agent or all-agents. (4) `backend-node/scripts/agent-eval-gate.ts` (~140 LOC) — CI gate: compares each agent's latest `results/<agent>.json` against its `baselines/<agent>.json`. Three failure conditions: mean < `PASS_THRESHOLD` (0.8), overall mean drop ≥ `REGRESSION_DROP_THRESHOLD` (0.05 = 5pp), ANY per-case drop ≥ 5pp (so a single regressed case can't hide behind unchanged overall mean). Treats `baselineMean === 0` as first-run / no-baseline-yet rather than as "lost all the points" — prevents the pre-T6.c placeholder baselines from triggering false failures. Prints per-agent breakdown with both the current mean and the drop vs baseline. Exit 0 = pass, 1 = fail, 2 = IO error. (5) **Blessed 4 baselines** with real scores from the T6 + T6.b runs: `alm_decision.json` mean 91.7% (10 cases, range 84–99%), `cfo_copilot.json` mean 89% (1 case), `committee_report.json` mean 88% (1 case), `risk_monitor.json` mean 93% (2 cases, both at 93%). Note: "T6.b initial bless after per-agent scorer generalization (2026-05-17)". (6) `backend-node/package.json` — three new npm scripts: `agent-eval`, `agent-eval:bless`, `agent-eval:gate`. Not wired into `lint` because they're slow (~5–10s each due to ts-node startup) and the lint chain runs on pre-commit; future CI workflow can invoke them separately. (7) `backend-node/src/__tests__/agent-eval-gate.spec.ts` (6 tests via subprocess `spawnSync` against the real CLI, ~150 LOC). Stages synthetic `results/test_regression_gate_synth.json` + `baselines/test_regression_gate_synth.json` files (the synthetic agentId is not in `script-registry.ts`, so the gate's no-argument default doesn't see it — only the test's explicit `--agent` invocation does). Six scenarios: first-run-no-baseline passes when mean ≥ threshold; first-run-no-baseline fails when mean < threshold; no drop passes; drop < 5pp passes (within tolerance); overall drop ≥ 5pp fails as regression; per-case drop ≥ 5pp fails even when overall mean is unchanged (the asymmetric-cases scenario — `case-a: 0.95→0.85`, `case-b: 0.85→0.95`, mean unchanged but case-a regressed). 60s jest timeout because ts-node startup is the bottleneck. afterAll cleanup is best-effort behind try/catch — some sandbox FS mounts disallow unlink for files the test runner created, and the synthetic agentId is invisible to production runs anyway. (8) `backend-node/.gitignore` — `/test/agent-evals/results/` and `/test/agent-evals/baselines/test_regression_gate_synth.json` are volatile, regenerated on every test run; the real `baselines/*.json` for production agents ARE committed (they're the blessed reference). **Why this matters for going-live:** with the T6.c pipeline in place, an agent-quality regression — whether from a prompt edit, a model bump, a tool registry change, or a scoring-adapter tweak — gets caught at CI time instead of after deployment. The 4 blessed baselines (alm 91.7%, cfo 89%, committee 88%, rm 93%) become the floor that future changes ratchet from. The 5pp tolerance gives room for normal score jitter; per-case detection prevents asymmetric regressions from hiding. **Coverage:** every script in `script-registry.ts` is now CI-gated end-to-end (scoring → baseline → drop detection). The gate spec locks the regression mechanism itself so a future refactor of `agent-eval-gate.ts` can't silently disable detection. **Operator workflow:** developer runs `npm run agent-eval` locally before pushing; CI runs `npm run agent-eval:gate` against the blessed baselines; on an intentional improvement, the developer runs `BLESS_REASON="..." npm run agent-eval:bless --agent <ID>` and commits the new baseline alongside the prompt/tool change. **Verification:** `jest src/__tests__/llm-script-scoring-006-010.spec.ts src/__tests__/llm-script-contracts-other-agents.spec.ts src/__tests__/agent-eval-adapters.spec.ts src/__tests__/agent-eval-gate.spec.ts` → 58/58 green in ~5s (8 + 20 + 24 + 6). `npm run agent-eval --quiet` → exit 0. `npm run agent-eval:gate` → all 4 agents 0.0pp drop, gate PASSED. `tsc --noEmit -p tsconfig.json` → no errors. `verify-no-orphan-spec` → 431 specs scanned, 4 skipped (each with integration-scope comment), 0 orphans.
